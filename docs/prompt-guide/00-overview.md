# Prompt 编写指南 · 总览

> 按项目生命周期分阶段，每个阶段对应一份指南。
> 手写时先判断"我现在在哪个阶段"，再套对应模板。

---

## 怎么用

1. 看下表找到你当前所处的阶段
2. 打开对应文件，找到匹配的场景
3. 复制模板，填入 `[方括号]` 内容即可

---

## 阶段总览

| 阶段         | 你的状态               | 指南文件            | 核心问题                     |
| ------------ | ---------------------- | ------------------- | ---------------------------- |
| **项目规划** | 还没写代码，方向未定   | `01-planning.md`    | 做什么？怎么做？用什么技术？ |
| **开发执行** | 方向已定，正在写代码   | `02-development.md` | 照着做、修 bug、重构         |
| **质量保障** | 代码写完，要保证没问题 | `03-quality.md`     | 审查、测试、文档、部署       |

## 各阶段场景速查

### 阶段一：项目规划（`01-planning.md`）

| 场景     | 什么时候用             | 你要写什么                 | AI 做什么            |
| -------- | ---------------------- | -------------------------- | -------------------- |
| 技术选型 | 几个方案里选一个       | 背景 + 候选方案 + 评估维度 | 对比 + 推荐          |
| 方案设计 | 模糊需求，要出完整方案 | 背景 + 初步想法            | 访谈 → 分析 → 出方案 |
| 需求拆解 | PRD 拿到了，要拆任务   | PRD + 技术栈               | 拆任务 + 标依赖      |
| 学习理解 | 不懂某技术 / 陌生代码  | 困惑点 + 代码              | 解释 + 举例          |

### 阶段二：开发执行（`02-development.md`）

| 场景     | 什么时候用         | 你要写什么               | AI 做什么           |
| -------- | ------------------ | ------------------------ | ------------------- |
| 明确执行 | 需求明确，照着做   | 目标 + 输入 + 约束       | 照做                |
| 排查诊断 | 报错 / 数据不对    | 现象 + 复现 + 已排查     | 定位根因            |
| 重构优化 | 代码能跑但有问题   | 现状 + 目标 + 不可动区域 | 分析 + 改动清单     |
| 数据迁移 | 改表 / 清洗 / 迁移 | 现状 + 目标 + 回滚方案   | 迁移脚本 + 风险评估 |

### 阶段三：质量保障（`03-quality.md`）

| 场景     | 什么时候用            | 你要写什么          | AI 做什么       |
| -------- | --------------------- | ------------------- | --------------- |
| 代码审查 | 写完了，帮看一眼      | 代码 + 关注点       | 找问题          |
| 测试编写 | 补测试 / 写 E2E       | 代码 + 覆盖要求     | 生成测试用例    |
| 文档生成 | 出 API 文档 / README  | 代码 + 受众 + 格式  | 生成文档        |
| 部署运维 | CI/CD / Docker / 发布 | 项目信息 + 目标环境 | 生成配置 / 脚本 |

## 编写原则（通用）

1. **先判断阶段**：规划阶段不要急着要代码，执行阶段不要让 AI 做选型
2. **写"要什么"而非"怎么做"**：你负责目标，AI 负责实现细节
3. **约束比需求更重要**：告诉 AI 不要做什么，比告诉它要做什么更有效
4. **给参考**：指定"参考现有 xxx 的写法"，比描述风格高效 10 倍
5. **不确定就访谈**：方案设计场景，让 AI 问你，而不是你猜它需要什么
6. **给上下文**：AI 不在你的项目里，技术栈、项目结构、历史包袱都要写
7. **分阶段交付**：复杂需求让 AI 分步输出（先分析 → 再方案 → 最后代码），不要一次性要结果

## 组合使用（跨阶段）

实际开发中，一个 Prompt 经常跨阶段。核心技巧：**把后面阶段的要求，前置写进当前 Prompt 的 `## 交付要求` 里**。

### 怎么写

在任意模板末尾追加 `## 交付要求`，写明完成后还要做什么：

```markdown
## 交付要求

- [自检 / 补测试 / 生成文档 / ...]
- 发现问题直接修复后再交付
- 最后列出你做了哪些额外工作
```

### 常见组合

| 组合               | 交付要求里加什么                                 |
| ------------------ | ------------------------------------------------ |
| 重构 + 自检        | "改完自检逻辑 bug 和行为变化"                    |
| 新功能 + 测试      | "写完同步补单测，参考现有测试风格"               |
| 新功能 + 文档      | "实现完后生成 API 文档"                          |
| 重构 + 自检 + 测试 | "改完自检，再补对应测试"                         |
| 数据迁移 + 自检    | "写完检查脚本边界情况，模拟异常输入验证回滚方案" |

### 备选：分两轮对话

如果组合太复杂，也可以拆成两轮：

1. 第一轮：用执行模板让 AI 完成主任务
2. 第二轮：直接说"现在对你刚写的代码做 self-review，检查逻辑 bug、边界遗漏、性能问题，发现问题直接修"
