# Prompt 指南 · 阶段二：开发执行

> 方向已定，正在写代码。需要 AI 帮你实现、修 bug、重构。

---

## 场景一：明确执行

> 需求明确，告诉 AI 要什么，让它照着做。

**适用**：新增接口、对接页面、写工具函数、加功能。

```markdown
# 需求：[标题]

## 目标

[做什么]

## 输入/接口

[接口地址、参数、文档链接]

## 要求

[代码放哪、参考什么]

## 约束

[技术栈、不要改什么]
```

---

## 场景二：排查诊断

> 出问题了，不知道根因在哪，让 AI 帮查。

**适用**：线上 bug、报错排查、数据异常、性能问题。

```markdown
# 问题：[现象]

## 现象

[什么场景、什么表现]

## 复现步骤

1. ...
2. ...

## 已排查

- [试过什么、排除了什么]

## 相关代码

- 文件路径:行号

## 期望

[正确行为应该是什么]
```

---

## 场景三：重构优化

> 代码能跑但有问题，需要改但怕改坏。

**适用**：性能优化、代码结构改善、技术债清理、拆分耦合。

```markdown
# 重构：[模块名称]

## 现状

- 代码位置：src/xxx/
- 当前问题：[慢 / 乱 / 耦合 / 不好维护]

## 目标

[改完后达到什么效果]

## 要求

- 先分析现有代码结构和问题
- 列出改动清单（改哪些文件、改什么）
- 标注风险点（哪些改动可能影响现有功能）
- 不要动 [xxx]，那是稳定的

## 约束

- 保持现有接口签名不变
- 改完要能通过现有测试
```

---

## 场景四：数据迁移

> 表结构变更、数据清洗、存量数据迁移。

**适用**：数据库改表、数据修正、存量迁移。

```markdown
# 数据迁移：[目标]

## 现状

- 数据库：[MySQL / MongoDB / ...]
- 表名：[xxx]
- 数据量：[约 xxx 万条]
- 当前结构：[字段说明]

## 目标

[迁移到什么结构 / 清洗什么数据]

## 要求

1. 先写迁移脚本
2. 标注可能失败的场景和回滚方案
3. 考虑迁移期间是否需要停服
4. 大数据量需要分批处理

## 约束

- 必须有回滚 SQL
- 不要在业务高峰执行
```

---

## 本阶段原则

- **目标要具体**：不要"帮我优化代码"，要"把 xxx 函数的响应时间从 2s 降到 200ms"
- **给文件路径和行号**：AI 需要知道改哪里，不要让它猜
- **说清楚不要动什么**：比说要改什么更重要
- **报错要给完整上下文**：错误堆栈、复现步骤、已排查范围，缺一不可
